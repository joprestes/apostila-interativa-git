🚀 Por que Git é essencial para QA?

Git é fundamental para profissionais de QA porque:

  • Permite versionamento de scripts de automação
  • Facilita colaboração entre QAs e desenvolvedores
  • Integra com pipelines de CI/CD para testes
  • Mantém histórico de mudanças em casos de teste
  • Permite rollback seguro em caso de problemas

Vamos revisar os conceitos essenciais! 🚀

🧠 Conceitos Essenciais

Mesmo que o grupo já utilize Git, é importante reforçar o porquê de cada prática no contexto de QA:

Conceito Importância no contexto de QA
Controle de versão Permite histórico das mudanças e rollback em caso de erro.
Branching Facilita trabalho paralelo (ex.: novas features, correções, ajustes de automação).
Pull Request (PR) Garante revisão e validação de qualidade antes do merge.
Commits semânticos Facilitam rastreabilidade e leitura do histórico do repositório.

🔍 Por que isso importa para QA?

Controle de versão é especialmente importante quando você está desenvolvendo automação de testes. Se um script quebra após uma mudança, você pode facilmente voltar à versão anterior que funcionava.

Branching permite que diferentes QAs trabalhem em paralelo - um pode estar criando testes para uma nova feature enquanto outro corrige bugs em testes existentes.

Pull Requests garantem que todo código de teste passe por revisão, mantendo a qualidade e compartilhando conhecimento entre a equipe.

Commits semânticos ajudam a entender rapidamente o que mudou, facilitando debugging e manutenção dos testes.

💡 Exemplo Prático

Imagine que você está trabalhando em testes automatizados para uma funcionalidade de login:

  • Você cria uma branch específica: test/validacoes-login
  • Faz commits semânticos: test: adicionar validação de senha inválida
  • Abre um PR para revisão da equipe
  • Se algo der errado, pode fazer rollback facilmente

💻 Comandos Básicos - Revisão Prática

Aqui estão os comandos essenciais que você usa no dia a dia como QA. Esta é uma revisão prática com foco nas situações mais comuns:

Ação Comando Observação
Clonar repositório git clone <url> Cria cópia local do repositório.
Criar nova branch git checkout -b feature/nome-da-feature Sempre criar branch a partir da main ou develop.
Ver status local git status Mostra alterações e arquivos pendentes de commit.
Adicionar mudanças git add . ou git add <arquivo> Adiciona arquivos para o próximo commit.
Criar commit git commit -m "feat: descrição do commit" Use prefixos semânticos (ver abaixo).
Enviar alterações git push origin feature/nome-da-feature Publica a branch no GitHub.
Atualizar branch local git pull origin main Atualiza código antes do merge.
Reverter commit git revert <hash> Cria novo commit revertendo o anterior.
Descartar mudanças locais git restore <arquivo> Cancela edições ainda não commitadas.

🎯 Fluxo Típico de Trabalho para QA

# 1. Clonar o repositório (primeira vez) git clone https://github.com/empresa/projeto-testes.git # 2. Criar branch para nova funcionalidade de teste git checkout -b test/validacao-login-api # 3. Verificar status das mudanças git status # 4. Adicionar arquivos modificados git add . # 5. Fazer commit com mensagem semântica git commit -m "test: adicionar validações de login com credenciais inválidas" # 6. Enviar para o repositório remoto git push origin test/validacao-login-api # 7. Atualizar branch local com mudanças da main git pull origin main

⚠️ Comandos Importantes para Situações de Emergência

# Reverter último commit (mantém mudanças locais) git reset --soft HEAD~1 # Descartar todas as mudanças locais não commitadas git restore . # Ver histórico de commits git log --oneline # Mudar para branch específica git checkout nome-da-branch # Ver todas as branches git branch -a

✅ Correto! O comando git add . adiciona todos os arquivos modificados, novos e deletados no diretório atual para a staging area. É muito útil quando você fez mudanças em múltiplos arquivos de teste e quer incluir todos no próximo commit.

✅ Correto! O comando git revert <hash> é a forma mais segura de desfazer um commit. Ele cria um novo commit que desfaz as mudanças do commit especificado, mantendo o histórico intacto. Isso é especialmente importante em ambientes de produção ou quando trabalhando em equipe.
❌ Incorreto. O comando correto é git revert <hash>. Este comando é mais seguro que reset porque cria um novo commit desfazendo as mudanças, ao invés de apagar o histórico.

❌ Incorreto. O comando correto é git pull origin main. Este comando combina fetch (baixar) e merge (integrar) das mudanças da branch main remota para sua branch local atual.

🧱 Padrão de Nomenclatura de Branches

Uma nomenclatura consistente de branches facilita a organização do trabalho e a colaboração em equipe. Vamos ver os padrões recomendados:

📋 Padrão Recomendado

<tipo>/<descrição-curta>

Exemplo: test/validacao-login-api

🏷️ Tipos Comuns para QA

Tipo Uso Exemplo
feature/ Nova funcionalidade ou fluxo de automação feature/adicionar-teste-login
fix/ Correção de bug ou ajuste no teste fix/ajuste-assert-login
test/ Melhoria ou novo teste automatizado test/api-validacoes-negativas
chore/ Alterações de configuração ou dependência chore/configurar-playwright
docs/ Atualização de documentação docs/instrucoes-de-execucao

✅ Boas Práticas

  • Sempre usar letras minúsculas e hífens (-).
  • Evitar nomes genéricos como feature/teste ou fix/bug.
  • Sempre relacionar a branch a uma tarefa do board (Jira/Trello/etc).

💡 Exemplos Práticos para QA

# ✅ Bons exemplos test/validacao-email-invalido feature/automacao-checkout-completo fix/timeout-teste-api-pagamento chore/atualizar-selenium-webdriver docs/guia-execucao-testes-mobile # ❌ Exemplos ruins test/teste feature/nova-feature fix/bug minha-branch teste123

🎯 Relacionando com Tarefas

Uma boa prática é incluir o número da tarefa na branch:

# Com número da tarefa do Jira test/QA-123-validacao-login-api feature/QA-456-automacao-carrinho-compras fix/QA-789-corrigir-seletor-botao-submit

🔍 Commits Semânticos

Adotar um padrão de commit ajuda na leitura do histórico e nas pipelines de CI/CD. É especialmente útil para QAs que precisam rastrear mudanças em testes.

📋 Formato Padrão

<tipo>: <descrição breve>

Exemplo: test: adicionar validação de login com credenciais inválidas

🏷️ Tipos Sugeridos para QA

Tipo Significado Exemplo
feat: nova funcionalidade feat: adicionar validação de login inválido
fix: correção de bug fix: corrigir timeout no teste de API
test: adição/ajuste de testes test: incluir cenários de erro 500
refactor: melhoria de código sem alterar comportamento refactor: extrair função de login
chore: manutenção e setup chore: atualizar dependências npm
docs: documentação docs: adicionar instruções de execução

💡 Exemplos Específicos para Automação de Testes

# Adicionando novos testes test: adicionar cenários de validação de CPF inválido test: incluir testes de API para endpoint de usuários test: criar suite de testes para funcionalidade de carrinho # Corrigindo testes existentes fix: corrigir seletor do botão de submit no teste de login fix: ajustar timeout para testes de API lenta fix: resolver flakiness no teste de upload de arquivo # Melhorias na automação refactor: extrair helper para autenticação de usuários refactor: organizar page objects em pastas por módulo feat: adicionar relatório HTML para resultados dos testes # Configuração e manutenção chore: atualizar Playwright para versão 1.40 chore: configurar pipeline de CI para testes E2E docs: atualizar README com instruções de execução local

🎯 Por que usar commits semânticos?

  • Histórico claro: Facilita entender o que mudou rapidamente
  • Automação: Ferramentas podem gerar changelogs automaticamente
  • CI/CD: Pipelines podem reagir diferente baseado no tipo de commit
  • Debugging: Mais fácil encontrar quando um bug foi introduzido
  • Colaboração: Equipe entende melhor as mudanças

🔄 Pull Request (PR) e Revisão de Código

🧠 Objetivo do PR

Garantir que todo código entregue:

  • Está alinhado ao padrão do time;
  • Passou pelos testes automatizados;
  • Foi revisado e validado por outro QA/dev.

💡 Boas Práticas ao Abrir um PR

📏 Tamanho e Foco

  • Mantenha o PR pequeno e focado.
  • Uma mudança lógica por PR.

📝 Título Descritivo

✅ Bom exemplo:

"Adicionar testes de login com credenciais inválidas"

❌ Exemplo ruim:

"Testes de login"

📋 Descrição Clara

Descreva claramente o que foi feito:

## O que foi feito - Adicionado teste automatizado de login inválido - Criado helper para autenticação ## Como testar - Rodar `npx playwright test --grep @login`

✅ Checklist antes de abrir PR

  • Adicione reviewers relevantes
  • Verifique que os testes e linters estão passando
  • Inclua instruções de como testar
  • Referencie a tarefa do board (Jira/Trello)
  • Verifique se a branch está atualizada com a main

🧩 Boas Práticas ao Revisar um PR

A revisão de código é uma oportunidade de aprendizado e melhoria contínua da qualidade. Veja como fazer revisões efetivas:

🎯 Princípios da Boa Revisão

  • Comente de forma construtiva e específica
  • Verifique claridade, legibilidade e padrões do time
  • Se necessário, faça o checkout local para testar (gh pr checkout <número>)
  • Evite "apenas aprovar" — toda revisão é uma oportunidade de aprendizado

✅ Comentários Construtivos

# ✅ Bom comentário "Seria interessante adicionar um timeout maior aqui, pois a API pode demorar mais em ambiente de produção. Que tal usar 10 segundos ao invés de 5?" # ❌ Comentário não construtivo "Isso está errado."

🔍 O que Revisar em Testes

  • Clareza: Os testes são fáceis de entender?
  • Cobertura: Os cenários importantes estão cobertos?
  • Seletores: Estão robustos e não frágeis?
  • Assertions: São específicas e claras?
  • Dados de teste: Estão bem organizados?
  • Padrões: Seguem as convenções da equipe?

🛠️ Testando Localmente

Para testar um PR localmente:

# Usando GitHub CLI gh pr checkout 123 # Ou manualmente git fetch origin pull/123/head:pr-123 git checkout pr-123 # Executar os testes npm test # ou npx playwright test

💬 Tipos de Feedback

  • Sugestão: "Que tal extrair isso para uma função helper?"
  • Pergunta: "Por que escolheu esse seletor específico?"
  • Elogio: "Ótima cobertura de cenários negativos!"
  • Bloqueante: "Este teste vai falhar em mobile, precisa ajustar"
  • Nitpick: "Pequeno detalhe: poderia usar const ao invés de let aqui"
